## Storage Architecture Overview

The Rotaract District 3192 ecosystem relies heavily on unstructured data: profile pictures, club logos, event banners, meeting audio, payment receipts, and generated PDF reports. 

**Primary Provider:** Supabase Storage (backed by S3).  
**Scale:** Designed to handle 20GB+ of annual growth seamlessly.  
**Security:** Supabase Storage natively integrates with PostgreSQL Row-Level Security (RLS) to ensure that users can only access private files they are authorized to see.

---

## Bucket Definition & Folder Hierarchy

Buckets represent the highest level of isolation. Do not put all files into a single bucket.

### 1. `public-assets` (Public Bucket)
Used for globally accessible media that requires no authentication. Highly cached at the CDN level.
* **Hierarchy:**
  * `/clubs/{club_id}/logo.{ext}`
  * `/events/{event_id}/banner.{ext}`
  * `/users/{user_id}/avatar.{ext}`
  * `/showcase/gallery/`

### 2. `private-documents` (Private Bucket)
Used for sensitive data requiring JWT authorization.
* **Hierarchy:**
  * `/moms/{club_id}/{mom_id}/audio.{ext}`
  * `/receipts/payments/{payment_id}.pdf`
  * `/certificates/{course_id}/{user_id}.pdf`

### 3. `system-exports` (Private Bucket)
Used for asynchronous background jobs generated by the Reporting Team.
* **Hierarchy:**
  * `/reports/{year}-{month}/district_summary.csv`
  * `/backups/database_dumps/`

## Naming Standards & Compression

### Naming Standards
* Files must never retain their original user-uploaded names to prevent path traversal attacks and encoding issues.
* **Standard Format:** `{uuid_v4}.{extension}` or `{entity_id}/{type}.{extension}`
* Example: Instead of `my_pic_final(1).jpg`, store as `avatar_e4b2a1...jpg`.

### Compression Rules
* All images must be compressed **client-side** before hitting the Supabase upload endpoint.
* **Images (Avatars/Logos):** Max 2MB. Converted to WebP or optimized JPEG.
* **Documents (PDFs):** Max 10MB.
* **Audio (MOMs):** Max 50MB. Enforce mono-channel, 64kbps MP3/M4A format to drastically save space without losing speech intelligibility.

## Public vs. Private Access (Security Policies)

Supabase Storage links bucket policies directly to PostgreSQL RLS policies in the `storage.objects` table.

### Public Access Policy
For the `public-assets` bucket, the RLS policy is completely open for reads:
```sql
CREATE POLICY "Public Access"
ON storage.objects FOR SELECT
USING (bucket_id = 'public-assets');
```

### Private Access Policy (MOM Audio)
For the `private-documents` bucket, a user can only read an audio file if they belong to the club that owns the MOM.
```sql
CREATE POLICY "Club Members can read MOM Audio"
ON storage.objects FOR SELECT
USING (
  bucket_id = 'private-documents' 
  AND auth.uid() IN (
    SELECT id FROM public.member_profiles 
    WHERE club_id = (storage.foldername(name))[2]::uuid -- Extracts {club_id} from path
  )
);
```

## Image Optimization & Thumbnail Strategy

Supabase Storage provides native Image Transformation via the URL.

* **Original Upload:** An event banner is uploaded once in high-resolution (e.g., 1920x1080).
* **Delivery (Thumbnails):** The Next.js frontend fetches the image using Supabase's transformation query parameters.
  * *Thumbnail:* `/storage/v1/object/public/public-assets/events/{id}/banner.jpg?width=300&height=300&resize=cover`
* **Next.js Image Component:** The frontend will utilize `next/image` pointing to the Supabase domain to ensure WebP delivery and lazy loading.
* **Benefit:** Saves massive amounts of storage by never explicitly saving thumbnail derivatives in the bucket.

## Data Flow Lifecycle

### 1. Upload Flow
1. Client requests an upload token/URL or authenticates via Supabase Client.
2. Client compresses the file.
3. File is pushed to Supabase Storage.
4. On success, the frontend calls the Node.js Core API to update the PostgreSQL table (e.g., `UPDATE member_profiles SET avatar_url = {url}`).

### 2. Download Flow
1. **Public:** Direct URL via Supabase CDN (e.g., Cloudflare).
2. **Private:** Client requests a **Signed URL** (valid for 60 seconds) from the Backend. The Backend verifies permissions, asks Supabase for the signed URL, and returns it to the client.

### 3. Delete Flow
1. When an entity is deleted, an asynchronous background job is triggered to purge the associated files from Supabase. 
2. Files are never deleted synchronously during API calls to avoid hanging requests if the storage provider is slow.

## DigitalOcean Migration Strategy

*Although the current focus is Supabase Storage, L1/L2 originally referenced DigitalOcean Spaces (S3-compatible). If a migration from DO Spaces to Supabase Storage is required:*

1. **Mapping:** Both use S3-compatible architecture. DO Spaces buckets map 1:1 to Supabase buckets.
2. **Transfer:** Use the open-source tool `rclone` to execute a server-to-server sync from `blr1.digitaloceanspaces.com` to the Supabase S3 endpoint.
3. **Database Update:** Run a SQL script to mass-replace the base URL string `https://rotaract.blr1.digitaloceanspaces.com` with `https://[SUPABASE_REF].supabase.co/storage/v1/object/public/` in all `logo_url` and `banner_url` columns.

## Retention & Disaster Recovery

### Retention Lifecycle
* **Active Data:** Kept in hot Supabase storage indefinitely.
* **Audio Transcripts (MOMs):** Kept for 3 years, then a background worker moves the audio file to a cheaper Google Cloud Storage Coldline bucket and deletes it from Supabase to save costs.
* **Deleted Users:** Avatars are purged immediately upon DPDP account deletion requests.

### Backup & Disaster Recovery (DR)
* **Strategy:** Supabase does not automatically backup storage objects in their standard daily DB backup.
* **DR Implementation:** An AWS Lambda (or cron worker) runs weekly to sync the entire `private-documents` and `public-assets` buckets to an isolated AWS S3 Glacier (or Google Coldline) vault.
* **Recovery Time Objective (RTO):** 24 hours to restore critical assets from Glacier.
* **Recovery Point Objective (RPO):** 7 days (maximum data loss acceptable for unstructured media).
