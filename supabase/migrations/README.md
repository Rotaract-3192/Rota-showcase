# Supabase Migrations

## Purpose
This folder stores all raw SQL migration files that mutate the PostgreSQL database schema. Migrations dictate the creation, alteration, and dropping of tables, views, indexes, and triggers.

## Migration Naming Conventions
To ensure execution in the exact intended sequence and maintain a clear history, all migration files must follow a strict naming convention:

`[ID]_[snake_case_description].sql`

**Where:**
- `[ID]` is a sequential timestamp generated by the Supabase CLI (e.g., `20260701123000`), ensuring chronologically accurate execution.
- `[snake_case_description]` clearly states what the migration is accomplishing.

### Examples
- `20260701000000_initial_schema.sql`
- `20260701000001_relationships.sql`
- `20260701000002_rls.sql`
- `20260701000003_indexes.sql`
- `20260701000004_seed_reference.sql`

## Important Rules
1. **Never edit an existing migration file** once it has been applied to production. Always create a new migration to alter the schema.
2. **Idempotency:** Ensure scripts can be run safely (e.g., use `CREATE TABLE IF NOT EXISTS` or standard Supabase CLI generated files).
3. **No App Data in Migrations:** Use the `seed/` directory for test data. Migrations should only contain schema changes or strictly static reference data required for the application to boot.
